Resolves #51.

Adds a vertical-domain (VisDrone, UAV small-object, 10 classes) **edge-deployment +
export-consistency** example for **YOLO-Master-EsMoE-N** under `examples/YOLO-Master-EsMoE-VisDrone-Edge/`:

- `configs/visdrone.yaml` — VisDrone2019-DET config.
- `scripts/train_visdrone.py` — fine-tune EsMoE-N (COCO → nc=10).
- `scripts/recalibrate_bn.py` — BN running-stat recalibration (fixes the sparse-eval BN drift).
- `scripts/export_models.py` — export ONNX (onnxslim+onnxsim) + MNN (+INT8 weight-quant) + NCNN.
- `scripts/export_ncnn_dense.py` — NCNN via dense full-softmax routing (pnnx can't lower MoE `topk`).
- `scripts/patch_dense_forward.py` — **export-consistency patch** (see below).
- `python/` — aspect-ratio letterbox, area-adaptive (small-object) NMS, unified ONNX+MNN+NCNN backends, COCO-style mAP50/50-95 eval, consistency driver, latency benchmark.
- `cpp/` — CMake C++ (ONNXRuntime) edge inference; verified on Linux x86_64 and Windows x86_64 (MSVC 19.44), with an ARM64 toolchain file + multi-arch Dockerfiles.

## Usage

```bash
# 0. apply the export-consistency patch to a YOLO-Master checkout
python scripts/patch_dense_forward.py path/to/YOLO-Master/ultralytics/nn/modules/moe/modules.py
# 1. fine-tune + BN recalibrate + export
python scripts/train_visdrone.py --epochs 50 --imgsz 640 --batch 16
python scripts/recalibrate_bn.py --src best.pt --dst best_recal.pt --device 0
python scripts/export_models.py --model best_recal.pt --imgsz 640
# 2. consistency (548 val) + C++ edge inference
cd python && python eval_consistency.py --pytorch ../best_recal.pt --onnx ../exports/best_recal.onnx --mnn ../exports/best_recal.mnn --imgsz 640 --num-classes 10
cd ../cpp && cmake -S . -B build -DONNXRUNTIME_ROOT_DIR=$(pwd)/third_party/onnxruntime && cmake --build build -j && ./build/yolo_edge ../exports/best_recal.onnx image.jpg --nc 10 --bench 200
```

## Export-consistency patch (`patch_dense_forward.py`) — why this exists

`ES_MOE` exposes two expert-combine paths:

- `_sparse_forward` (eager eval) computes only the top-k experts **and** applies a
  `dynamic_threshold=0.4` pruning — weak experts are dropped.
- `_dense_forward` is **forced during ONNX/TorchScript tracing** (the sparse path has
  data-dependent control flow tracing can't follow) and keeps **all** experts.

Because the pruning is not replicated in `_dense_forward`, **the exported graph produces
different (worse) features than eager eval**, so exported mAP collapses to ~0 even though
the PyTorch model reaches 6% mAP50-95.

`patch_dense_forward.py` edits `ES_MOE._dense_forward` so that, **during export only**, it
replicates `_sparse_forward`'s pruning with traceable ops (`topk` / `one_hot` / `argmax`).
After the patch the exported ONNX/MNN graph matches eager eval to ~1e-4 and recovers the
full mAP. The training path is unchanged.

> This is the **export-side** counterpart of the **eval-side** `--no-sparse-eval` flag from
> #81 — both address the same sparse/dense mismatch; this patch makes the *exported* model
> consistent with the model's mAP.

**No core library change is required** — `patch_dense_forward.py` is a standalone script the
user runs against their YOLO-Master checkout; the example does not modify `ultralytics/`.

## Results (VisDrone val, 548 images, imgsz=640)

Consistency (issue #51 target ΔmAP50-95 < 0.5%):

| Backend | mAP50 | mAP50-95 | ΔmAP50-95 vs PyTorch |
|---------|-------|----------|----------------------|
| PyTorch (native) | 12.09% | 6.00% | — |
| ONNX (opset 17) | 12.08% | 5.99% | **−0.004%** ✅ |
| MNN | 12.08% | 5.99% | **−0.006%** ✅ |
| NCNN (dense) | — | ~0% | — (MoE pruning can't lower to NCNN; see `export_ncnn_dense.py`) |

Edge latency/throughput (same image → identical 33 detections across platforms):

| Backend / platform | mean (ms) | FPS |
|--------------------|-----------|-----|
| C++ ONNXRuntime — Linux x86_64 | 53.0 | 18.9 |
| C++ ONNXRuntime — Windows x86_64 (MSVC 19.44) | 83.6 | 12.0 |
| MNN (Python, CPU) | 56.1 | 17.8 |
| NCNN (Python, CPU) | 100.0 | 10.0 |

NCNN note: pnnx can't lower the MoE routing's `topk`/comparison ops (they become no-op
layers NCNN refuses to register), so the pruned path is not representable in NCNN. The
example exports a dense (no-topk) NCNN graph that loads and runs, at the cost of accuracy.
ONNX/MNN are the recommended edge paths and both meet the <0.5% target with non-zero mAP.
